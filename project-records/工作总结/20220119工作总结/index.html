2022/01/19工作总结 #

协同开发 #

协同开发主要体现在如下方面：

  • 代码管理
  • 项目进度管理
  • 汇报

代码管理 #

代码管理主要是通过git，常用命令如下：

  1. 首次获取代码
bash
git clone ~~~
  1. 创建新分支

在远端根据master创建新分支，在本地拉取分支即可

bash
git fetch origin newBranch
git checkout newBranch
  1. 提交代码
bash
git add .
git commit -m "~~~"
git push
  1. merge到test-master时候发生冲突

当发生冲突时应该在本地解决

bash
# 切换到目标分支
git checkout test-master
git pull origin test-master
git merge my-dev

在vscode查找git冲突位置【<<<<】，选中哪方代码。选中完成后检查代码，然后提交。 5. 共用开发分支，获取代码发生冲突

bash
# 先获取，再合并
git pull origin my-dev
  1. 检测代码

本地检查：

bash
git diff

远端检查：通过merge request预提交检查 6. 撤销在开发但未提交的代码

bash
git stash
  1. 撤销已提交的代码

先通过git log查看提交记录，然后通过git revert ~~~撤销

项目进度管理 #

系统设计 #

主要为需求分析和原型设计两部分

需求分析：即弄清楚每个地方的细节要做的逻辑功能

原型设计：即弄清楚每个地方的细节要做的样式及交互效果

一般情况下是通过撰写的文档确定

开发 #

开发过程即写代码

对接 #

对接主要有三种场景：

  • 对接设计师
  • 对接后端
  • 对接测试
对接设计师 #

设计师给出原型图，按照该原型制作即可

设计师给出图标，多为SVG文件，缺少直接索要即可

设计师在项目进度中的话语权最低

对接后端 #

接口联调就是和后端给出的接口进行对接，在项目中获取的数据正确

在后端给出接口前要尽可能的把单纯的前端功能做完整，然后通过后端给出的接口调整数据格式，尽量适配原先数据

一般情况下，功能是前后端同时开发，但给出接口的时机不同：

  • 后端在开发前给出接口文档，前后端独立开发，最终按照约定即可
  • 后端开发完成后给出接口文档，前端适配

第一种情况好办，即使要求传入数据发生修改，也不会太大；

第二种前端开发就比较难受，一些特定字段类型修改会大大影响数据的处理流程，因此这时候有三种处理方式，处理成本依次递增：

  1. 设法让后端修改接口数据格式，减少自身工作量
  2. 前端在service层或model层手动编写函数适配数据格式
  3. 大改系统数据处理流程
对接测试 #

一般情况下，当前后端都开发和联调完成后，正式测试前会有评审流程，主要展示测试给出的Case。之后各自将代码提交到测试环境，测试通过设计的Case挨条对系统测试。

如果系统功能较多，某些功能问题较多或未开发完成，采用本地开发机方式

代码依次提交：本地--->测试--->预发--->线上

测试环境：(主要是功能性问题)

  • 功能是否完全
  • 交互是否有误
  • 数据交互是否正确

预发环境：(主要是需求性问题)

  • 数据集是否缺少或错误

测试过程中出现的大部分问题，责任分割：

  • 前端：样式交互问题、请求报文
  • 后端：响应报文

除此之外，一些特殊问题需要前后端讨论协商解决

对接领导 #

当系统开发完成和大体测试完成后，需要在周会上进行展示，展示有时会给出意见，这时候不需要问啥，改就行了

汇报 #

项目开发还收到会议进度影响，每周五下午组会，每两周一或周四晚上双周会，汇报最近的工作状况

根据汇报进度，可以确定工期模式：

小需求：

  • 2天开发，1天联调，1天测试，1天上线和汇报；共1周
  • 3天开发，2天联调，2天测试，1天上线和汇报；共1周半

中需求：

  • 1周开发，4天联调，1周测试，1天汇报；共3周

项目管理 #

数据抽离 #

数据分类 #

状态数据：模块自身状态state及模块影响props。状态数据呈现树形状态，和React组件对应。

动态数据：从后端获取的需要动态获取然后展示或是影响配置的一些数据，不受到生命周期影响，也不会因关闭应用而丢失。动态数据由外部注入，呈平行状态，数据之间可以联动，但是平等的。

数据管理 #

根据对系统数据的分类，可以进行管理

  1. 事件通知

事件的监听和触发机制，onClick、onSelect等一系列监听，可以随意的定义触发时机和地点，每一个事件的触发对应一个监听，关系是一一对应。

  1. 共享对象

让不同组件引用共同的数据源，具体方式有三种：

  • 公共父组件
  • Context API
  • redux或dva等状态管理库

在产品树需求中，接口返回的动态数据产品树、产品列表因为要在系统太多地方使用，因此被单独共享出来。

使用共享对象，尽量只进行获取操作，小心修改、删除操作，如果有需要注意：

  • 共享的数据发生修改对各处的影响如何
  • 共享的数据是否是互斥访问的，虽然JS是单线程，但也需要注意

抽出公共代码 #

  1. CSS样式。可以将某些CSS样式如字体、颜色、背景单独提出来，写到style文件夹下，每次使用时候倒入即可。
  2. 常量。如文案、环境路径等单独写到constants文件夹下，在需要的地方倒入即可。
  3. 函数。处理http、upload、url、waterMark等常常反复使用的逻辑单独提出来。
  4. 组件封装。
